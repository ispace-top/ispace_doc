## 1. 引言

### 1.1 产品概述

**爱思文档（i·Space Doc）** 是一款私有化部署的企业级文档管理与知识库平台。系统基于 Django 4.2 框架构建，核心理念是 **"一切皆文档"**——借鉴 Linux "一切皆文件" 的哲学，系统中的所有内容单元都是文档，知识组织通过文档间的归属关系自然形成，无需引入额外的容器概念。

**核心价值主张**：

- 私有化部署，数据完全自主可控
- 一切皆文档：用统一的文档模型承载知识组织、权限管理、协作编辑
- 多层级组织架构与细粒度权限管理，适配中大型团队协作
- 企业级多认证体系：本地密码 / OIDC / 企业微信 / LDAP / 钉钉
- 丰富的文档编辑与交互体验（Markdown IR、划词评论、@提及、拖拽排序）
- 开箱即用的 5 步安装引导，降低部署门槛
- 暖色调深色/亮色双主题，CSS Custom Properties 驱动

**用户画像**：

- 主要用户群体为 10-500 人的中小型团队（技术团队优先，兼顾非技术部门）
- 典型场景包括：技术文档管理、团队知识库、项目文档协作、内部 API 文档、企业合规文档管理

### 1.2 核心理念：一切皆文档

> 借鉴 Linux 操作系统"一切皆文件"的哲学，i·Space Doc 将"一切皆文档"作为核心设计原则。

**传统 Wiki 的做法**：引入独立的"文集/空间/项目"容器概念。用户需要先创建空间，再在其中创建文档。概念负担重，操作路径长。

**i·Space Doc 的做法**：系统只有一种东西——文档。`parent_doc` 字段是唯一的组织手段。

- **文档即目录**：任何文档都可以拥有子文档。就像 Linux 中目录文件既是文件也能容纳文件，一篇文档既是可编辑的内容页，也可以是子文档的父节点——两者不互斥，可以同时兼具。
- **树形归属**：文档通过 `parent_doc` 指向父文档。`parent_doc=0` 的文档就是文档树的入口——不是什么特殊类型的实体，仅仅是没有父节点而已。
- **统一操作**：创建、编辑、删除、移动、授权——对所有文档无差别，无论它在树的什么位置。

```
# 一个技术团队的知识库

技术文档 (parent_doc=0)
├── 后端 (parent_doc=技术文档)
│   ├── Django框架指南 (parent_doc=后端)
│   └── API设计规范 (parent_doc=后端)
├── 前端 (parent_doc=技术文档)
│   ├── React实战教程 (parent_doc=前端)
│   └── TypeScript手册 (parent_doc=前端)
└── 运维 (parent_doc=技术文档)
    └── Docker部署指南 (parent_doc=运维)
```

每篇文档都可以：

- 承载正文内容（作为"页面"）
- 拥有子文档（作为"目录"）
- 同时兼具两者（既有内容又组织子文档）

### 1.3 术语定义

| 术语 | 英文标识 | 说明 |
| ---- | -------- | ---- |
| 用户 | User | 系统最小身份单位，一个登录账号对应一个用户 |
| 文档 | Doc | 系统唯一的内容实体。既是可编辑的"页面"，也是可容纳子文档的"节点"。`parent_doc=0` 仅仅表示该文档没有父节点 |
| 子文档 | Child Doc | 某文档的下级文档，`parent_doc` 指向父文档 ID |
| 分组 | Group | 单层扁平结构，由若干用户组成，用于批量授权和管理 |
| 组织架构 | Organization | 多层树状结构（公司→部门→小组），一个用户可归属多个节点 |
| 物化路径 | Materialized Path | 用字符串路径（如 `1/3/7/`）表示组织树节点位置，便于快速子树查询 |
| 文档授权 | Doc Permission | 对指定文档设置 view/edit/admin 权限的过程 |
| 有效权限 | Effective Permission | 用户对某文档经多路合并后的最终权限 |
| 软删除 | Soft Delete | 标记 `is_deleted=True` 而非物理删除，数据保留可恢复 |
| 划词评论 | Inline Comment | 在文档正文中选中文字后创建的评论，与选中文字锚定关联 |
| OAuth 绑定 | OAuth Binding | 第三方认证平台用户与本地用户的关联记录 |
| 目录同步 | Directory Sync | 从企业微信/LDAP 拉取组织架构和用户数据到本地的过程 |
| 水印 | Watermark | 文档正文区域的文字或图片水印，保护文档安全 |
| 访问密码 | Access Password | 公开文档的可选密码保护，输入正确密码后才能查看 |
| WebHook | Webhook | 文档事件（创建/更新/删除/评论/点赞）的 HTTP 回调通知 |

### 1.6 版本管理规则

#### 版本号格式

i·Space Doc 采用 **三位或四位版本号**（`x.xx.xx` / `x.xx.xx.xxx`）表述形式：

```
主版本号 . 次版本号 . 修订号 [ . 补发号 ]
  (必须)     (必须)    (必须)    (非常规)
```

- **常规发布**使用三位版本号
- **第四位**为非常规版本号，仅在第三位版本发布后出现紧急问题且需要立即修复并补发时启用

#### 各位含义

| 位次 | 名称 | 递增条件 | 典型变更 |
|:----:|------|----------|----------|
| **第一位** | 主版本号 | 重大需求版本更新、重大模块上线、架构重构、产品方向变更 | v0→v1 正式版发布；架构重构；认证体系上线 |
| **第二位** | 次版本号 | 功能级需求新增、模块重构 | 新增绘图/云存储模块；组织架构重构；新认证后端接入 |
| **第三位** | 修订号 | 问题修复、体验优化、日志迭代、小功能调整 | Bug 修复；UI 微调；性能优化；依赖升级；文档更新 |
| **第四位** | 补发号 | 第三位版本发布后出现紧急问题，需立即修复并补发 | `0.9.1` 中的严重 Bug，紧急补发 `0.9.1.1` |

#### 递增规则

1. **第一位递增** — 第二、三位归零。如 `0.9.7` → `1.0.0`
2. **第二位递增** — 第三位归零。如 `0.9.7` → `0.10.0`
3. **第三位递增** — 不影响其他位。如 `0.9.7` → `0.9.8`
4. **第四位递增** — 不影响前三位。如 `0.9.7` → `0.9.7.1` → `0.9.7.2`
5. 第四位版本之后若正常发布第三位迭代，第四位自然消除：`0.9.7.2` → `0.9.8`

#### 版本演进示例

| 版本号 | 释义 |
|--------|------|
| `0.9.0` | **开发预览版（当前版本）** |
| `0.9.1` | 预览版首次迭代修复 |
| `0.9.1.1` | 针对 `0.9.1` 的紧急补发 |
| `0.10.0` | 第二个功能级迭代 |
| `0.10.3` | 功能迭代后的第 3 次修复 |
| `1.0.0` | 首个正式发布版本 |

#### Git 分支与版本对应

| 分支类型 | 命名示例 | 对应版本 | 说明 |
|----------|----------|:--------:|------|
| `main` | — | 最新稳定版 | 生产环境代码，每个发布提交打 Tag |
| `feature/v0.10` | `feature/v0.10-drawing` | `0.10.x` | 功能分支，合并后第二位递增 |
| `fix/` | `fix/login-error` | `x.x.N` | 修复分支，合并后第三位递增 |
| `hotfix/` | `hotfix/v0.9.1.1` | `x.x.x.N` | 紧急补发分支，合并后第四位递增 |

> **版本 Tag 规范**：在 `main` 分支的发布提交上打 Tag，格式 `v0.9.0`（三位常规）或 `v0.9.1.1`（四位补发）。

## 2. 用户体系

### 2.1 用户模型

系统基于 Django `django.contrib.auth.models.User`，并通过 `UserProfile` 扩展用户档案。

**Django 内置 User（基础层）**：
`username`、`password`、`email`、`first_name`、`last_name`、`is_active`、`is_superuser`、`date_joined`、`last_login`

**UserProfile（扩展档案层）**：

| 字段 | 类型 | 约束 | 说明 |
| ---- | ---- | ---- | ---- |
| `user` | OneToOneField → User | CASCADE | 与内置 User 一一对应 |
| `avatar` | ImageField | null，上传至 `avatars/%Y/%m/` | 用户头像，1:1 裁剪 200×200 |
| `gender` | CharField(1) | M/F/U，默认 U | 性别 |
| `phone` | CharField(20) | unique，null | 手机号，全局唯一 |
| `bio` | TextField(512) | 默认空字串 | 个性说明，最长 200 字符 |
| `last_active` | DateTimeField | null | 最后活跃时间，登录时自动更新 |
| `notify_settings` | TextField | 默认 `'{}'` | 通知偏好 JSON |
| `wecom_userid` | CharField(128) | blank，db_index | 企业微信 UserID |
| `dingtalk_userid` | CharField(128) | blank，db_index | 钉钉 UserID |

**自动创建**：通过 Django Signal (`post_save`) 在 User 创建时自动创建 UserProfile。

### 2.2 用户角色体系

| 角色 | 判定方式 | 权限边界 |
| ---- | -------- | -------- |
| 普通用户 | 所有已登录用户 | 浏览公开文档，参与评论，编辑有权限的文档 |
| 分组管理员 | `Group.owner_id == user.id` | 管理所属分组的增删改、成员增减、管理员转让 |
| 部门管理员 | `OrgNode.admin_id == user.id`（含祖先节点传递） | 管理所属部门及下级部门的人员增减、部门结构 |
| 组织管理员 | 任一祖先节点的 `admin_id == user.id` | 管理整棵组织架构树、人员、管理员任命 |
| 系统管理员（超管） | `user.is_superuser == True` | 拥有全部权限，对一切文档隐式 admin 权限 |

**超管特殊规则**：

1. 对**所有文档**拥有隐式 `admin` 权限，无需 `DocPermission` 表记录
2. 在文档权限管理界面中**不可见、不可操作、不可移除**
3. 最少管理员约束**不统计超管**
4. 超管的敏感操作（编辑、删除）需记录审计日志

### 2.3 注册与登录

**注册**（`/register/`）：
- 必填：用户名（5-20 位字母数字，大小写不敏感唯一）、邮箱、密码（≥6 位）
- 两种模式：开放注册 / 邀请码注册（`RegisterCode` 含使用次数上限和过期日期）

**登录**（`/login/`）：
- 认证方式：Django `authenticate`（username + password）
- 失败锁定：连续 5 次失败后锁定 15 分钟
- 登录记录：每次尝试写入 `LoginRecord`（用户名、IP、UA、成功/失败、失败原因）
- 重定向安全：登录后 URL 通过 `is_internal_path` 校验，防止开放重定向漏洞

**密码管理**：
- 存储：Django PBKDF2 哈希（SHA-256，360,000 次迭代）
- 修改密码：需验证旧密码，新密码 ≥ 6 位
- 忘记密码：通过邮箱验证码重置

### 2.4 个人中心

**路由**：`/user/center/?tab=<tab_name>`

**功能子页**：

| Tab | 功能说明 |
| --- | -------- |
| 个人资料 | 头像上传裁剪 + 姓名/性别/手机/个性说明 |
| 账号安全 | 修改密码 + 最近 20 条登录记录（IP/设备/时间） |
| 通知设置 | 8 类通知邮件开关 + 每日汇总时间 + 第三方渠道绑定 |
| 我的分组 | 分组管理面板：创建/查看/管理成员/退出分组 |
| 我的文档 | 已发布文档分页列表，含草稿数量角标 |
| 文档模板 | 模板管理面板：新建/编辑/删除模板 |
| 我的组织 | 所属组织树展示，高亮主属部门 |
| 浏览记录 | 基于持久化存储的浏览历史，支持分页 |
| 我的收藏 | 收藏的文档列表 |
| 回收站 | 已删除文档的恢复/彻底删除 |

## 3. 认证体系

### 3.1 认证架构

iSpaceDoc 支持多种企业级认证方式，统一通过 `config.ini` 配置管理，登录页自动展示已启用的认证入口。

```
登录页 (/login)
    │
    ├── 本地密码表单 → Django auth.authenticate()
    │
    └── OAuth 按钮区
        │  fetch /auth/providers/ → 根据 config.ini 返回已启用列表
        │
        ├── redirect 类型 (OIDC/WeCom/DingTalk)
        │   → /auth/<provider>/login/ → 跳转 IdP → /auth/<provider>/callback/ → 登录
        │
        └── form 类型 (LDAP)
            → 弹出用户名密码表单 → POST /auth/<provider>/login/form/ → 登录
```

**后端架构**：

```
backend/apps/doc/auth/
├── base.py          AuthBackend 抽象基类，定义 get_authorize_url / authenticate 接口
├── config.py        从 config.ini 读取配置，构建后端实例
├── views.py         统一 OAuth 视图（login / callback / sso / bind / providers）
├── urls.py          路由定义
├── oidc.py          OIDC 认证后端
├── wecom.py         企业微信认证后端
├── ldap.py          LDAP 认证后端
├── dingtalk.py      钉钉认证后端
└── adapter.py       用户创建/绑定适配器
```

**认证提供者发现**：`GET /auth/providers/` 返回已启用的认证方式列表，前端根据返回结果动态渲染登录页 OAuth 按钮区。`enabled` 字段支持在不丢失配置的情况下临时关闭某个后端。

### 3.2 本地密码认证

- 标准 Django `authenticate(username, password)`
- 支持登录验证码（`enable_login_check_code` 配置）
- 支持全站强制登录（`RequiredLoginMiddleware`）
- 支持关闭注册（`close_register` 配置）

### 3.3 OIDC 单点登录

**支持 IdP**：Keycloak / Auth0 / Azure AD / Okta / 其他标准 OIDC Provider

**配置项**（`[auth.oidc]`）：

| 配置键 | 说明 | 示例 |
| ------ | ---- | ---- |
| `enabled` | 是否启用 | `true` |
| `provider_name` | 显示名称 | `Keycloak` |
| `discovery_url` | OIDC Discovery URL | `https://keycloak.example.com/realms/myrealm/.well-known/openid-configuration` |
| `client_id` | 客户端 ID | `ispace-doc` |
| `client_secret` | 客户端密钥 | `xxx` |
| `scope` | 请求的 scope | `openid profile email` |

**认证流程**：
1. 用户点击 "OIDC 单点登录" → `/auth/oidc/login/`
2. 服务端构建 Authorization URL（含 nonce + state）→ 302 重定向到 IdP
3. 用户在 IdP 完成认证 → IdP 回调 `/auth/oidc/callback/?code=xxx&state=xxx`
4. 服务端用 code 换 token → 验证 id_token（签名/iss/aud/exp/nonce）→ 获取 userinfo
5. 通过 `adapter.py` 匹配或创建本地用户 → Django login() → 跳转首页

### 3.4 企业微信认证

**配置项**（`[auth.wecom]`）：

| 配置键 | 说明 |
| ------ | ---- |
| `enabled` | 是否启用 |
| `corp_id` | 企业 ID |
| `corp_secret` | 应用 Secret |
| `agent_id` | 应用 AgentId |

**两种登录方式**：
- **扫码登录**：构造企业微信 OAuth URL → 用户扫码 → 回调获取 code → 换取 userid → 匹配/创建用户
- **免登**（自建应用内）：通过 `code` 参数调用 `/auth/wecom/sso/`，在企业微信客户端内免密登录

### 3.5 LDAP 目录认证

**配置项**（`[auth.ldap]`）：

| 配置键 | 说明 | 示例 |
| ------ | ---- | ---- |
| `enabled` | 是否启用 | `true` |
| `server_uri` | LDAP 服务器地址 | `ldap://ldap.example.com:389` |
| `bind_dn` | 绑定 DN | `cn=admin,dc=example,dc=com` |
| `bind_password` | 绑定密码 | `admin_password` |
| `user_base_dn` | 用户搜索基 DN | `ou=users,dc=example,dc=com` |
| `user_filter` | 用户搜索过滤器 | `(uid=%(user)s)` |
| `username_attr` | 用户名属性 | `uid` |
| `email_attr` | 邮箱属性 | `mail` |
| `use_tls` | 是否启用 TLS | `false` |

**两种绑定模式**：
- **搜索模式**：先用 `bind_dn` 绑定 → 在 `user_base_dn` 下搜索匹配 `user_filter` 的用户 → 用搜索到的用户 DN + 用户输入的密码重新绑定
- **直接绑定模式**：用配置的 DN 模板直接替换 `%(user)s` 后绑定，无需搜索步骤

**交互**：登录页点击"LDAP 域账号登录"→ 弹出模态框输入用户名+密码 → POST `/auth/ldap/login/form/` → 认证成功建立 Session

### 3.6 钉钉认证

- OAuth 2.0 扫码登录：构造钉钉授权 URL → 用户扫码 → 回调 → 匹配/创建用户
- 免登支持（钉钉自建应用内）

### 3.7 目录同步

支持从企业微信和 LDAP 定时同步通讯录至本地组织结构：

**企业微信同步**（`sync_wecom_contacts`）：
- API 拉取部门列表（递归）→ 创建/更新 `OrgNode`（含 `external_source='wecom'` 标识）
- API 拉取部门成员列表 → 创建/更新 `User` + `OrgUser` + `UserProfile.wecom_userid`
- 支持 `--dry-run` 预览模式

**LDAP 同步**（`sync_ldap`）：
- 搜索组织单位（OU）→ 创建/更新 `OrgNode`（含 `external_source='ldap'` 标识）
- 搜索用户条目 → 创建/更新 `User` + `OrgUser`
- 支持 `--dry-run` 预览模式

**同步触发方式**：
- 管理后台认证配置页面手动触发
- 管理命令：`python manage.py sync_wecom_contacts` / `python manage.py sync_ldap`
- cron 定时任务（生产环境推荐）

### 3.8 用户绑定机制

OAuth 登录用户通过 `IspOAuthBinding` 模型记录绑定关系：

| 字段 | 说明 |
| ---- | ---- |
| `provider` | 认证后端标识（oidc/wecom/dingtalk） |
| `provider_user_id` | 第三方用户唯一 ID |
| `provider_user_name` | 第三方用户名 |
| `bound_at` | 绑定时间 |

**匹配策略**（按优先级）：
1. 通过 `IspOAuthBinding` 查找已有绑定 → 返回对应用户
2. 通过邮箱匹配已有用户 → 自动创建绑定 → 返回用户
3. 通过用户名匹配已有用户 → 自动创建绑定 → 返回用户
4. 以上均不匹配 → 自动创建新用户 + 绑定

| 字段 | 类型 | 说明 |

## 6. 文档体系

### 6.1 文档数据模型

```python
# Doc — backend/apps/doc/models.py
```

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| `name` | CharField(255) | 文档标题 |

**数据库索引**：

- `(sort)` — 排序查询
- `(is_deleted, deleted_at)` — 软删除过滤
- `(is_public)` — 公开文档过滤

### 6.2 文档即目录：层级组织

i·Space Doc 不设独立的"目录"或"文件夹"概念。任何文档都可以拥有子文档，从而自然扮演"目录"的角色。文档的两种能力可以共存：承载内容和拥有子文档。

### 6.3 文档编辑器

系统支持 3 种编辑器模式：

| 模式 | 编辑器 | 说明 |
| ---- | ------ | ---- |
| 2 | Vditor（默认） | 现代 Markdown IR 所见即所得 |
| 3 | iceEditor | 富文本编辑器 |
| 4 | Luckysheet | 在线电子表格 |

> 模式 1（Editormd）已废弃。

**内容存储流程**：
1. 编辑内容 → `pre_content`（Markdown 源码或 HTML）
2. 服务端渲染 → `content`（HTML）
3. 展示：Markdown 通过 marked.js 渲染；富文本直接展示；表格 Luckysheet 渲染

**Callout 块提示**：`> i/w/e/s/t` 前缀的 blockquote 自动渲染为彩色提示块（信息/提醒/关键/关注/技巧），图标通过 CSS `::before` 伪元素渲染。

**ECharts 图表**：6 种预置图表模板（柱状图/折线图/饼图/散点图/雷达图/仪表盘），插入 `echarts` fenced code block。

**Markdown 表格操作**：插入弹窗（行列数配置）+ IR 模式浮动工具栏（上方/下方插入行、左侧/右侧插入列、删除行/列/表格）。

### 6.4 文档树与导航

- 文档通过 `parent_doc` 构建树形层级，`parent_doc=0` 的文档为树入口节点
- 手风琴式展开：同级节点中同时最多只有一个节点的子文档列表展开
- 面包屑：显示从顶层到当前文档的完整路径，支持溢出隐藏
- 上下文导航：上一篇/下一篇（按 sort 值排序）

### 6.5 文档拖拽排序

| 操作 | 触发方式 | 效果 |
| ---- | -------- | ---- |
| 同级排序 | 拖拽到两个同级条目之间 | 插入当前位置，更新 sort |
| 移动为子文档 | 拖拽到目标文档上方悬停 600ms | 目标自动展开，可放入子级列表 |
| 取消拖拽 | 拖到无效区域或按 Escape | 条目回原位 |

**排序值策略**（间隙插入）：初始值 1000, 2000, 3000...；相邻间隙 < 10 时触发全量重排。

**约束校验**：操作者需对源文档和目标父文档均有编辑及以上权限；目标不可为源文档自身或其子孙文档。

### 6.6 文档历史版本

每次修改文档时自动保存历史快照（`DocHistory`），支持版本列表查看、左右双栏 diff 对比、回滚操作。

### 6.7 文档删除与恢复

**软删除机制**：标记 `is_deleted=True`，禁止物理删除。级联删除：删除父文档时子孙文档同步标记删除。

**删除确认分级**：
| 子文档数 | 确认级别 | 交互 |
| -------- | -------- | ---- |
| = 0 | 普通确认 | 确认对话框 |
| > 0 | 高危警告 | 红色标题 + 子文档列表 + 图形验证码校验 |

### 6.8 右侧大纲导航

文档保存时服务端预生成大纲 JSON（`Doc.outline` 字段），前端渲染层级目录结构。支持收起态（右侧标签条）+ 展开态（260px 面板），IntersectionObserver 滚动跟踪，点击条目平滑跳转。

### 6.9 文档草稿管理

| status | 含义 | 可见性 |
| ------ | ---- | ------ |
| 0 | 草稿 | 仅创建者本人可见 |
| 1 | 已发布 | 按权限规则可见 |

- 新建文档提供"保存草稿"和"发布"两个操作按钮
- 编辑器每 10 秒自动保存，带 `is_auto_save='1'` 标识（不写入 DocHistory）
- 首页展示"继续编辑"草稿区块（最近 5 条）
- 侧边栏"我的文档"链接显示草稿数量角标

### 6.10 文档模板

- 模板存储在 `DocTemp` 表（名称 + 内容 + 创建者）
- 管理入口：个人中心 → 文档模板 Tab
- 创建文档时从模板选择器一键套用

### 6.11 文档水印

**新增功能**。文档级水印控制，从 Doc 模型字段驱动：

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| `is_watermark` | BooleanField | 是否启用水印 |
| `watermark_type` | IntegerField | 1=文字水印，2=图片水印 |
| `watermark_value` | CharField(250) | 用户自定义文字，为空时默认使用用户名 |

**渲染效果**：文字水印在文档正文区域以 -15° 倾斜角度重复平铺，透明度 0.06-0.1，`pointer-events: none` 不干扰文本选择，打印时也可见。

### 6.12 访问密码保护

**v1.0 新增功能**。文档级访问控制：

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| `is_public` | BooleanField，default=True | 是否允许公开访问 |
| `access_password` | CharField(128)，blank | 非空时启用密码验证 |

- `is_public=False`：仅授权用户可访问
- `is_public=True` + `access_password` 非空：公开但需密码验证
- `is_public=True` + `access_password` 为空：完全公开

## 7. 文档权限系统

### 7.1 权限类型

| 权限级别 | 标识 | 允许操作 |
| -------- | ---- | -------- |
| 可见 | `view` | 查看文档内容、查看评论 |
| 可编辑 | `edit` | 查看 + 编辑文档内容、发表评论、拖拽排序 |
| 管理员 | `admin` | 查看 + 编辑 + 标记删除 + 修改权限设置 |

### 7.2 数据模型

**DocPermission**：

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| `doc` | FK → Doc | 目标文档 |
| `target_type` | CharField(10) | user / group / org |
| `target_id` | PositiveIntegerField | 授权对象 ID |
| `permission` | CharField(10) | view / edit / admin |
| `apply_to_children` | BooleanField | 是否同步到所有子文档 |
| `granted_by` | FK → User | 授权人 |
| `granted_at` | DateTimeField | 授权时间 |

### 7.3 有效权限计算引擎

```
最终权限 = MAX(
    用户直接授权,
    用户所属所有分组的权限,
    用户所属所有组织节点及祖先的权限
)
```

**特殊规则**：
1. 超管直接返回 `admin`
2. 文档创建者自动具有 `admin` 权限
3. 若文档及其祖先均未设置任何权限限制 → 所有登录用户默认 `view`

### 7.4 权限缓存

- 缓存 Key：`perm:{user_id}:{doc_id}:v{version}`
- 缓存 TTL：5 分钟
- 版本号 Key：`permver:{doc_id}`（变更时递增使旧缓存失效）
- **主动失效触发点**：授权/修改/撤销权限、分组成员变更、组织成员变更

### 7.5 约束与边界

1. **最少管理员约束**：每篇文档至少保留一个非超管的管理员
2. **子节点同步**：`apply_to_children=True` 时遍历所有子孙文档
3. **批量授权**：支持一次性添加多个授权条目
4. **权限申请**：无权限用户可申请，24 小时内同一文档不重复发送

## 8. 评论系统

### 8.1 普通评论

**数据模型（DocComment）**：支持嵌套回复（`parent` 自引用），软删除（`is_active`），@提及（`mentioned_users` M2M）。

**展示规则**：顶级评论按时间倒序；回复嵌套缩进最多 3 层；每条显示头像/用户名/相对时间/内容；已删除评论显示占位。

### 8.2 划词评论

**数据模型（InlineComment）**：`anchor_start` / `anchor_end` 字符偏移 + `anchor_hash`（MD5，用于文档编辑后重定位）+ `selected_text`（≤500 字符）。

**交互流程**：选中文字 → 浮动操作条 → 点击"评论" → 高亮下划线 + 输入框 → 提交后气泡数字 +1。

### 8.3 @提及功能

- 评论/编辑器中输入 `@` 触发用户搜索下拉（前 20 条）
- 服务端正则提取 @用户名，存储到 `mentioned_users`
- 文档正文 @提及在编辑保存时对比新旧内容，仅通知新增的 @用户

## 19. 安全需求

| 层面 | 措施 |
| ---- | ---- |
| 密码存储 | Django PBKDF2 哈希（SHA-256，360,000 次迭代） |
| 登录保护 | 5 次失败锁定 15 分钟；登录记录全量入库（IP/UA/时间） |
| 重定向安全 | 登录后跳转 URL 使用 `is_internal_path` 白名单校验 |
| 权限校验 | **服务端强制执行**，前端权限展示仅做 UI 控制 |
| CSRF | POST/PUT/DELETE 端点启用 CSRF 中间件 |
| XSS | `jsonXssFilter` 对所有 JSON 响应 HTML 实体转义 |
| SQL 注入 | Django ORM 参数化查询 |
| 文件上传 | 扩展名白名单 + 大小限制 + Zip bomb 检测 |
| 敏感配置 | 数据库密码、邮件授权码使用 `enctry/dectry` 加密存储 |
| SECRET_KEY | 安装引导时生成，长度 ≥ 50 字符 |
| 部署安全 | 生产关闭 DEBUG、HTTPS 强制、uWSGI + Nginx |
| WebHook 签名 | HMAC-SHA256 请求签名验证，防止伪造回调 |
| 访问密码 | 使用 Django PBKDF2 哈希存储，不存明文 |
| OAuth State | OIDC/WeCom/DingTalk 使用 cryptographic nonce + state 防 CSRF |
